Grundlagen

Was ist XProc?

  • XML-basierte Programmiersprache für (XML-basierte) Datenverarbeitung
  • Abstraktion geläufiger Verarbeitungsschritte in steps, die zu pipelines kombiniert werden

  • Spezifikationen

    • Version 1.0 W3C Recommendation (2010)
    • Version 3.0 W3C Community Group Report (2022)

  • Implementierung

    • MorganaXProc-III
    • XML Calabash 3 (under development)

Anwendungsszenarien

  • Publikationspipelines

    • Verarbeitung von XML-Dokumenten in unterschiedliche Zielformate (single source publishing)
    • Kombination von Transformationen, Validierung und anderen Verarbeitungsschritten
    • Beispiel: transpect.io

  • Datenpflege

    • Korrektur und Pflege von XML-Dokumenten
    • Werkwerfkorrekturen

Warum XProc 3.0?

  • XPath 3.1 & XQuery and XPath Data Model 3.1
  • Verarbeitung von nicht-XML-Dokumenten
  • vereinfachte Sytnax
  • aktive Community

Terminologie – Ports, Dokumente, Pipelines

Port

Ein Port ist ein Eingang (p:input) bzw. Ausgang (p:output) eines Schrittes. Ein Schritt kann beliebig viele Eingänge und beliebig viele Ausgänge besitzen. Er besitzt jedoch höchstens einen primären Eingang und höchstens einen primären Ausgang.

Schritte sind über Ports miteinander verbunden.

Dokument

Ein Dokument ist ein abgeschlossener Inhalt eines bestimmten Typs. Dokumente haben Eigenschaften, zum Beispiel und namentlich einen Inhaltstyp (content type) der sie z.B. als XML-Dokument (application/xml) oder Binärdokument (application/octet-stream) ausweist. Der Inhaltstyp eines Dokuments entscheidet über die anwendbaren Schritte.

Pipeline

Eine Pipeline ist eine (nicht-lineare) Abfolge von atomischen Verarbeitungsschritten. Eine Pipeline kann selbst als Schritt fungieren.

Implizite Verbindung von Ports

Der primäre Eingabeport der Pipeline ist mit dem primären Eingabeport des in Dokumentreihenfolge ersten Schritts verbunden.

Der primäre Ausgabeport des in Dokumentreihenfolge letzten Schritts ist mit dem primären Ausgabeport der Pipeline verbunden.

Folgen zwei Schritte in Dokumentreihenfolge aufeinander und die Verbindung der Ports ist nicht explizit, dann wird der primäre Ausgabeport des ersten Schritts mit dem primären Eingabeport des zweiten Schrittes verbunden.

Implizite Verbindung von Ports

Implizite Verbindung von Ports – Zusammenfassung

In vielen Fällen ist es ausreichend, sich auf die Regeln der implizten Verbindung von Ports zu verlassen.

Die explizite Verbindung von Ports wird spätestens dann notwendig, wenn…

  • eine Pipeline ein Dokument in zwei unabhängigen Schritten verarbeitet
  • mehrere Eingabeports eines Schrittes benutzt werden
  • mehrere Ausgabeports eines Schrittes benutzt werden

Explizite Verbindung von Ports

Dokumente werden gezogen (pull), nicht geschoben (push).

  • Verbindung mit einem externen Dokument

    <p:with-input port="…" href="…"/>

  • Verbindung mit dem Ausgabeport eines anderen Schritts

    <p:with-input port="…" pipe="port@schritt"/>

Explizite Verbindung von Ports

  • Verbindung mit einem inline Dokument

    <p:with-input port="…">
  <p:inline>
    <document>…</document>
  </p:inline>
</p:with-input>

  • Verbindung mit der leeren Sequenz

    <p:with-input port="…">
  <p:empty/>
</p:with-input>

Explizite Verbindung von Ports

Explizite Verbindung von Ports – Zusammenfassung

Die Reihenfolge in der Schritte ausgeführt werden ist durch die Verbindung ihrer Ports definiert, nicht durch die Reihenfolge in der Pipeline. Die Regeln der impliziten Portverbindung sorgen dafür, dass Schritt einer einfachen Pipeline in Dokumentreihenfolge ausgeführt werden.

Schritte, die nicht miteinander verbunden sind werden in keiner bestimmten Reihenfolge ausgeführt.

p:delete

The p:delete step deletes items specified by a selection pattern from the source input document and produces the resulting document, with the deleted items removed, on the result port.

Modifikation eines einzelnen Dokuments (04-single-document)

p:insert

The p:insert step inserts the insertion port's document into the source port's document relative to the matching elements in the source port's document.

Definition eines eigenen Schritts

Mit p:declare-step werden nicht nur ganze Pipelines, sondern auch Schritte innerhalb einer Pipeline definiert. Die drei Verarbeitungsschritte aus der vorangegangenen Übung können als ein Schritt zusammengefasst werden.

Eigene Schritte werden als Typen definiert die durch einen QName identifiziert werden.

PS: Und ja, eine komplette Pipeline beliebiger Komplexität kann als Schritt importiert werden.
